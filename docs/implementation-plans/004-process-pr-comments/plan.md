# 004: Process PR Comments Plugin (`pprc`)

<!-- execution-mode: manual -->
<!-- scope: phase -->

> A new dogfooded plugin (`pprc`, skill `/process-pr-comments`) that pulls a pull request's review comments from GitHub, classifies each as **Fixed / Won't-fix / Needs-clarification**, optionally edits code + commits + pushes a fix, then posts a per-comment reply to the PR thread. **Both the code push and every reply are approved by a human before they take effect** (diff/commit approval before push; approve / edit / skip before each reply). GitHub access uses the **current user's auth**: `gh auth token` supplies the token once, then **all** GitHub calls — PR detection, comment fetch, replies, and the unresolved-thread GraphQL query — go directly to `api.github.com` via `Invoke-RestMethod`. The skill is **interactive-only** (no headless mode). Packaged per the **Plan 002** plugin-registry conventions and installed into skalary's own `.github/` (dogfood).

## Decisions

- **Plugin identity.** Single plugin named `pprc` (process-pr-comments), invoked as `/process-pr-comments`. One bundle: a workflow skill `skills/process-pr-comments/SKILL.md` plus a PowerShell module `skills/process-pr-comments/scripts/GitHubPr.psm1` (GitHub I/O + git ops). Slug matches repo convention (`cr`, `dr`, `cip`, `ci`, `cdn`, `udn`).
- **Built on Plan 002.** Packaging follows the registry layout: canonical source under `plugins/pprc/` (mirror-structured), `plugin.json`, validated by `schemas/plugin.schema.json`, installed/dogfooded via `Install-Plugin.ps1` / `Sync-Dogfood.ps1`. **Plan 002 is a hard prerequisite for dogfood packaging (Phase 5)**; Phases 1–4 author code under `plugins/pprc/` and test it directly without requiring the registry tooling to exist.
- **Auth = current user token via `gh auth token`.** `Get-GitHubToken` shells `gh auth token` once (the only use of `gh`), captures the token in-memory, and uses it as the `Authorization: Bearer` header. Preflight checks `gh --version` + `gh auth status` and fails with actionable guidance (run `gh auth login`) if `gh` is missing or unauthenticated. The token is **never** written to disk or logged; redacted in verbose output. No PAT stored, no Credential Manager.
- **All API calls via `Invoke-GitHubRest` / `Invoke-GitHubGraphQL` against `api.github.com`.** `Invoke-GitHubRest` sends `Invoke-RestMethod` requests with the Bearer header + `Accept: application/vnd.github+json`: detect PR (list base-repo open PRs, see PR detection below), list inline review comments (`GET /repos/{o}/{r}/pulls/{n}/comments`), review summaries (`GET /repos/{o}/{r}/pulls/{n}/reviews`), and post replies. It captures response headers on **both** success (`-ResponseHeadersVariable`) and failure (from the `HttpResponseException.Response` in PS7) so rate-limit/`Link` headers are available even on a thrown 403. `Invoke-GitHubGraphQL` POSTs `{query,variables}` to `https://api.github.com/graphql` to fetch `pullRequest.reviewThreads { isResolved, comments }` because **REST does not expose review-thread resolved state**. **Both wrappers paginate fully**: REST follows the `Link` `rel=next` header; GraphQL follows cursor pagination (`pageInfo { hasNextPage endCursor }` with `first`/`after`) for `reviewThreads` **and** the nested `comments` connection. Rate limiting is detected on `403`/`429` via `X-RateLimit-Remaining: 0` (primary) **and** a `Retry-After` header (secondary/abuse limits, which need not zero the remaining count); both abort with the reset/retry time. No tight retry loop.
- **Comment scope.** Inline review comments **and** review summary bodies. Inline comments are filtered to **unresolved threads only** (via GraphQL `isResolved=false`). Review summary bodies (top-level review `body`) have no thread state → always included as informational context. **Out of scope:** general issue/conversation comments (`/issues/{n}/comments` as a *source*), though issue-comment posting is used as the reply channel for non-inline replies, and a narrow read of the **authenticated user's own** issue comments is permitted **solely for summary-reply dedup** (see below).
- **Thread granularity (not per-comment).** `GET /pulls/{n}/comments` returns both thread-root comments and replies. The unit of processing is the **review thread**, not the individual comment: items are grouped by thread (via the GraphQL `reviewThreads` set joined to REST comments by `databaseId`), the **thread root comment** is the processing unit, and its id is the only valid `in_reply_to` target. The root is selected **order-independently** as the REST member comment whose `in_reply_to_id` is `null` (not the GraphQL connection's first node, whose order is lost across nested-comment pagination). Replying to a mid-thread reply id is invalid, so the root id is always used. This also prevents duplicate replies into one thread.
- **Target PR detection (fork-aware).** GitHub's `head=` filter keys on the **head** repo owner, which is wrong for the fork workflow (where `origin` is the fork and the PR lives on upstream). Instead: list open PRs on the **base** repo (`GET /repos/{o}/{r}/pulls?state=open`, paginated) and filter by `head.ref == currentBranch`. Require exactly one match; error clearly on zero or multiple (multiple = same branch name across forks). Classify `head.repo.full_name != base.full_name` (or `head.repo == null` for a deleted fork) as **cross-repository** and refuse explicitly in the branch-safety guard. No `-PrNumber` override (out of scope; can add later).
- **Branch safety (strict, explicit target).** Before any commit/push: refuse if the PR is cross-repository (fork head / null head repo) or if the local HEAD branch != PR `headRefName`. The push target is **explicit** — `HEAD:refs/heads/{headRefName}` to the remote whose push URL matches the base repo `head.repo.full_name` — never a bare `git push` that follows arbitrary upstream config. Never force-push (repo rule). `git push` authenticates via git's ambient credential helper (independent of the REST Bearer token).
- **Dispositions (3).** Each processed thread is classified `Fixed` | `WontFix` | `NeedsClarification`. `Fixed` → edit code, commit, **(diff approval)**, push, reply with commit SHA + summary. `WontFix` → reply explaining why it does not apply / is intentional, no code change. `NeedsClarification` → reply asking the reviewer a question, no code change. (Defer-to-follow-up-issue is **out of scope**.)
- **Commit grouping = agent decides.** The agent groups related thread fixes into one commit and separates unrelated ones. Each reply references the SHA(s) that addressed it. Commits stage **only explicit paths from the agent's edit set** (no `git add -A`). A pre-existing dirty worktree is snapshotted first; files with pre-existing uncommitted user changes are refused/isolated unless explicitly approved, so the workflow never sweeps a user's in-progress edits into a fix commit.
- **Two human-approval gates (interactive-only).** (1) **Before push:** the skill presents the full diff + commit set; the human approves once before any `git push`. (2) **Before each reply:** the skill presents the target (thread root id / PR), disposition, and full reply text and asks **approve / edit / skip** (via `vscode_askQuestions`); edit lets the human rewrite the body; only approved replies post; skip leaves the thread untouched. Deterministic **two-phase** order: **Phase A (code)** — fix + commit *all* `Fixed` threads (agent grouping), present the full combined diff + commit set, approve **once**, then push once. **Phase B (replies)** — per thread compose → approve / edit / skip → post, where a `Fixed` reply posts **only after** its commit's push has succeeded; a failed push aborts that thread's reply (left for the next run). **No reply is posted and no commit is pushed without explicit human approval.** The skill is **interactive-only**: it requires an interactive session for both gates and does not define a headless/automated path (documented limitation).
- **Posting channel.** Inline reply → `POST /repos/{o}/{r}/pulls/{n}/comments` with `in_reply_to=<thread-root-comment-id>`. Reply to a review-summary body or any non-threaded item → `POST /repos/{o}/{r}/issues/{n}/comments` (PR conversation), quoting the original for context.
- **Threads left open.** The script does **not** resolve review threads (no GraphQL `resolveReviewThread`). The reviewer resolves manually after reading the reply. (Explicit user choice; keeps GraphQL surface read-only.)
- **Partial-failure semantics + re-run dedup.** Each thread is processed independently; no cross-thread transaction. A post/push failure on one thread does not roll back others. On re-run, dedup is based on the authenticated identity (`GET /user`, fetched once) **plus** a hidden marker token embedded in each posted reply body (an HTML comment `<!-- pprc:thread:<key> -->`, where `<key>` is the thread root comment id for inline replies and `summary-<reviewId>` for summary replies). For inline replies the marker is found by scanning the current user's replies in `pulls/{n}/comments` (in scope); for **summary** replies, which post to `issues/{n}/comments`, dedup reads the current user's **own** issue comments only (the narrow exception noted above) — best-effort if that read is unavailable. Any `<!-- pprc:` … `-->` sequence appearing in **quoted reviewer text** is stripped/encoded before composing, so a reviewer cannot forge a marker and silently suppress a thread (see Security).
- **PowerShell conventions.** `#requires -Version 7.0`; PSScriptAnalyzer zero warnings; module functions use approved verbs; `${var}` before literal colons; never assign `$host`; no leading `&`; invoke `.ps1`/module directly; Pester 5 with mocked **`Invoke-RestMethod`** (no live GitHub calls in tests).
- **Security.** Token kept in-memory only, never written/logged, redacted in verbose output; parse `owner`/`repo` from the trusted git remote with a parser that captures the **full** final path segment and strips only an optional trailing `.git` (so dotted repo names like `my.repo` are not truncated); PR number + `in_reply_to` coerced to integers; **reviewer comment text is untrusted at every ingestion point** (classify, fix, compose) — the agent treats it strictly as data, never follows operational instructions embedded in it, never interpolates it into a shell/git command, and limits fixes to the review's technical substance; commands derive from workflow/codebase context, not reviewer text; any `pprc` marker sequence (`<!-- pprc:` … `-->`) embedded in reviewer text is stripped/encoded before that text is quoted into a reply, preventing dedup-marker spoofing; strict explicit-target push guard; no `iex`/dot-source of any remote content.
- **Observability.** Structured console progress (`Write-Host`/`Write-Verbose`): PR resolved (number+url), counts (fetched / unresolved / by-disposition), per-comment disposition + commit SHA + posted reply id. Token never appears in any log line.

## Notes

- **Prerequisite:** GitHub CLI (`gh`) installed and authenticated (`gh auth login`) for the current user. This is the sole auth mechanism — no PAT, no Credential Manager.
- **Interactive-only:** the skill requires an interactive session for both approval gates (diff-before-push, reply approve/edit/skip). There is no headless/automated mode; this is a documented limitation, not a fallback.
- **Depends on Plan 002** for registry packaging/dogfood (Phase 5 only). Phases 1–4 are self-contained under `plugins/pprc/` + `tests/pprc/`.
- **Explicitly out of scope:** resolving review threads, opening/merging PRs, approving reviews, issue/conversation comments as a *source*, defer-to-follow-up-issue disposition, `-PrNumber` override, multi-PR batch, headless execution, live-GitHub integration tests.

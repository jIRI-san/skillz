> Consolidates **all** autopilot artifacts into a single self-contained `autopilot` **plugin**. Today autopilot is split: the plugin payload is the agent file only (`plugins/autopilot/agents/autopilot.agent.md`), while the runtime (`scripts/autopilot/**`), schema (`schemas/autopilot.schema.json`), and container context (`.devcontainer/autopilot/**`) live outside the plugin and are provisioned by a separate `Initialize-Autopilot.ps1`. This plan moves the scripts, schema, devcontainer, and config templates **into** `plugins/autopilot/`, packages them as ordinary plugin payload that installs under `.github/skills/autopilot/`, **adds a new internal autopilot SKILL.md** that owns the three autonomous execution modes (host / container / sandbox), implements the **custom host launch command** seam, adds a **first-run `.autopilot.json` bootstrap** to the skill, and **removes `Initialize-Autopilot.ps1`** (plugin install replaces it). `/ci`'s mode menu collapses to **Approve / Autopilot / Autonomous**; *Autonomous* hands off to the new skill.

## Scope Expansion (supersedes prior plan)

This plan was originally "Autopilot Skill Extraction" (extract autonomous modes into a skill + add a custom host command). Plan 002 (plugin registry) is now **delivered and archived**, and the repo/project was **renamed**. The user redirected the scope:

- **Autopilot becomes one self-contained plugin.** Scripts, schema, devcontainer, and config templates move from repo-root infra into `plugins/autopilot/` and install as plugin payload under `.github/`.
- **`ci`-plugin skill ownership reversed.** The new SKILL.md ships inside the **`autopilot`** plugin (co-located with the scripts it drives), not the `ci` plugin. `ci` keeps its existing `autopilot` dependency.
- **Plan 002 is in scope.** This plan edits the autopilot `plugin.json`, regenerates `registry.json`, re-syncs dogfood `.github/`, removes `Initialize-Autopilot.ps1`, and rewrites the plugin-registry "Autopilot Plugin vs Infra Split" narrative.

The prior DR Round 1 / Round 2 history (and the Plan 001 reconciliation) is retained in `evolution-log.md` for continuity but is partly superseded; see the **Restructure** entry there.

## Reconciliation with delivered code (verified by reading sources)

- **Real launcher signature** (`scripts/autopilot/launch.ps1`): `launch.ps1 -PlanSlug <slug> -Mode whole-plan|next-phase -Runtime host|container|sandbox [-Branch <branch>]`. Preserved verbatim; only the **path** changes (now invoked from the installed `.github/skills/autopilot/scripts/launch.ps1`).
- **Sandbox is fully implemented** (`scripts/autopilot/launch-sandbox.ps1`) — an available mode.
- **Host per-phase invocation** lives in `launch-host.ps1` `Invoke-CopilotPhase`, which hardcodes `Get-Command copilot` then branches at invocation time (`.bat`→`cmd.exe /c`, `.ps1`→`powershell.exe -File`, else direct exe) building `$psi.Arguments` **strings**. **No `.cmd` branch today** (npm global shims on Windows are `*.cmd`). This is the seam for the custom host command.
- **Plugin install model** (`schemas/plugin.schema.json`): `files[]` entries are `{src,dest}`; `src` under `plugins/<name>/`, `dest` under the consuming repo's `.github/`. The path-traversal guard rejects any `dest` outside `.github/`. No schema change needed — all new payload nests under `.github/skills/autopilot/` (agent stays `.github/agents/`).
- **Dogfood model** (`plugin-registry.design.md`): `plugins/` is authoritative; `.github/` is generated install output synced by `Sync-Dogfood.ps1`. Scripts therefore **run only from the installed `.github/skills/autopilot/scripts/` location**, never from `plugins/`.
- **Existing references to `scripts/autopilot/`** that must move with the scripts: `.editorconfig`, `.gitattributes`, `PSScriptAnalyzerSettings.psd1`, `.vscode/settings.json`, `.devcontainer/autopilot/Dockerfile` (COPY), and `Initialize-Autopilot.ps1` (being removed).
- **Schemas are JSON Schema draft 2020-12**; the new `autopilot.host.schema.json` matches.
- **Lint/format tooling exists** (`PSScriptAnalyzerSettings.psd1` + `.editorconfig`); **no Pester harness yet** (this plan adds the first).

## Decisions

### Plugin structure & layout

- **Single self-contained `autopilot` plugin.** Final source layout under `plugins/autopilot/`:
  - `agents/autopilot.agent.md` (exists)
  - `skills/autopilot/SKILL.md` (new — the autonomous-mode skill)
  - `scripts/*.ps1` + `scripts/container-entrypoint.sh` (moved from `scripts/autopilot/`)
  - `schemas/autopilot.schema.json` + `schemas/autopilot.host.schema.json` (moved + new)
  - `devcontainer/Dockerfile` (moved from `.devcontainer/autopilot/`; leading-dot dropped — it is build context, not VS Code's repo-root `.devcontainer`)
  - `.autopilot.json.example` + `.autopilot.host.json.example` (config templates)
- **Install layout — all `dest` under `.github/`:** bundle root is `.github/skills/autopilot/`:
  - agent → `.github/agents/autopilot.agent.md` (unchanged dest)
  - skill → `.github/skills/autopilot/SKILL.md`
  - scripts → `.github/skills/autopilot/scripts/…`
  - schemas → `.github/skills/autopilot/schemas/…`
  - devcontainer → `.github/skills/autopilot/devcontainer/…`
  - templates → `.github/skills/autopilot/.autopilot.json.example`, `.github/skills/autopilot/.autopilot.host.json.example`
  - Confinement guard (`dest` under `.github/`) is **unchanged** — no `plugin.schema.json`/install-flow change required.
  - **Manifest `dest` values are `.github`-relative** (`_Common.ps1` joins `<root>/.github` + `dest`): write `agents/autopilot.agent.md`, `skills/autopilot/SKILL.md`, `skills/autopilot/scripts/<file>`, etc. The `.github/…` paths above denote install *locations*, not literal `dest` strings — a literal `.github/` prefix would double-nest to `.github/.github/…` and trip the guard. (Filesystem paths the skill *invokes*, e.g. `.github/skills/autopilot/scripts/launch.ps1`, are real and keep the `.github/` prefix.)
- **Scripts run from the installed `.github/skills/autopilot/scripts/` location only** (dogfood or consumer). The live launchers already resolve repo root via `git rev-parse --show-toplevel` (location-independent, worktree-safe — `.git` may be a *file* in host-mode worktrees, which `git rev-parse` handles natively), so **no custom resolver is introduced** — keep `git rev-parse`. Only **sibling bundle-file references** (schemas, devcontainer, templates) change: resolve them **relative to `$PSScriptRoot`**. `Resolve-HostCommand` reads `<git-toplevel>/.autopilot.host.json` using the same `git rev-parse` root so it never disagrees with the launchers. Scripts are copied verbatim; no per-install customization.
- **`ci` plugin keeps its `autopilot` dependency** (already declared). The new SKILL.md ships in the **`autopilot`** plugin, reversing the prior DR1-#3 `ci`-ownership decision (now obsolete because scripts + skill co-locate in `autopilot`).
- **`Initialize-Autopilot.ps1` removed.** Plugin install (`Install-Plugin.ps1`) + dogfood `Sync-Dogfood.ps1` are the sole provisioning paths. All references to the script and to the old `scripts/autopilot/**` infra path are removed/updated.

### Config (`.autopilot.json`) & first-run bootstrap

- **`.autopilot.json` is per-repo user config at the repo root, gitignored.** The plugin ships `.autopilot.json.example` only; the real file is never committed and never plugin payload.
- **First-run bootstrap lives in the in-editor autopilot SKILL.** When `/ci` hands off to *Autonomous*, the skill checks for `.autopilot.json`; if absent it **interviews the user** (runtime, auth target, git provider/auth, build/test commands, model, timeout) and writes `.autopilot.json` from `.autopilot.json.example` **before** invoking `launch.ps1`, then **structurally validates** the generated file (required fields present + correct types, mirroring `launch.ps1`'s own hand-rolled field checks — not JSON-Schema validation, which `Test-Json` cannot do for draft 2020-12) and fails loud if invalid.
- **Headless `launch.ps1` does not interview.** It only validates `.autopilot.json` presence/shape and **fails loud with an actionable message** if missing (cannot prompt headlessly).

### Autonomous-mode skill

- **Skill identity:** `plugins/autopilot/skills/autopilot/SKILL.md`, name `autopilot`. Same name as the `autopilot` **agent** — VS Code distinguishes by type. The agent remains the per-phase executor invoked by `launch.ps1`; the skill is the in-editor mode-selection + bootstrap + orchestration entry point.
- **Internal-only, consumed as a textual include (not a skill invocation).** Frontmatter: `user-invocable: false`, `disable-model-invocation: true`. With both invocation paths disabled, `/ci` **reads it by path** and follows its steps inline. No `context: fork` (would imply skill-invocation semantics). No `/autopilot` slash command.
- **`/ci` menu collapses to 3 options:** Approve · Autopilot · **Autonomous**. Choosing *Autonomous* makes `/ci` read `.github/skills/autopilot/SKILL.md` and follow it; the skill presents the Host/Container/Sandbox sub-menu, the branch follow-up, the first-run bootstrap, and the launcher invocations.
- **`AUTOPILOT_CONTAINER=true` suppresses *Autonomous*** in `/ci` (already inside an autonomous container → only Approve/Autopilot offered).
- **Launcher invocations preserved verbatim** against the delivered signature, only re-pathed to the install location: `.github/skills/autopilot/scripts/launch.ps1 -PlanSlug <slug> -Mode whole-plan -Runtime host|container|sandbox [-Branch <chosen-branch>]`. Container/Sandbox keep the "Start from which branch? (Current / main)" follow-up, passed via `-Branch`. Host uses its own `feature/<slug>` worktree (no follow-up).

### Custom host launch command

- **Host mode may run a custom Copilot CLI executable** (e.g. a company wrapper injecting MCP servers + internal skills) instead of vanilla `copilot`.
- **Config:** operator-provisioned, gitignored `.autopilot.host.json` at repo root — separate from `.autopilot.json`. Fields: `command` (string — exe name or full path), `args` (string[] — static extra args appended to each per-phase invocation). No `displayName` (Host menu label is static).
- **Resolution vs. invocation are separate concerns.** Extract a `param()`-less helper `plugins/autopilot/scripts/host-command.ps1` exporting `Resolve-HostCommand`: reads `.autopilot.host.json` once, validates, resolves `command` to an absolute path, classifies type (`exe`/`bat`/`cmd`/`ps1` by extension), returns `@{ Path; Type; ExtraArgs }`. `launch-host.ps1` dot-sources it and calls it once before the phase loop, replacing the hardcoded `Get-Command copilot`. `Invoke-CopilotPhase` keeps its per-type invocation branching, **adds the missing `.cmd`→`cmd.exe /c` branch**, and appends `ExtraArgs` with **deterministic quoting**: the direct-`.exe` branch uses `ProcessStartInfo.ArgumentList` (no shell, safe tokenization); the `.bat`/`.cmd`→`cmd.exe /c` and `.ps1`→`powershell.exe -File` branches build the `$psi.Arguments` string with explicit per-token quoting (denylist-backed). No wholesale rewrite of the shell-wrapped branches' string approach. The **absent-config default also classifies `copilot`'s type by extension** (npm shims are `*.cmd`) — never hardcoded `exe`.
- **State handling:** absent → default `copilot` with no extra args (strict no-op vs today); **present-but-invalid** (malformed JSON, schema violation, empty `command`, or shell metacharacter in `command`/`args`) → **fail loudly before any phase starts** — never silent-fallback.
- **Validated by `plugins/autopilot/schemas/autopilot.host.schema.json`** (draft 2020-12).
- **Host-only.** `Resolve-HostCommand` is dot-sourced only by `launch-host.ps1`; container/sandbox orchestrators never read the file.
- **Static Host menu label.** The skill shows a fixed "Host autopilot" label and **does not read `.autopilot.host.json`** (preserves the "launcher is sole reader" invariant + the agent no-read rule).

### Security model (custom command — defense in depth)

- **No interactive approval for the host launcher (honest scope).** `launch-host.ps1` runs the Copilot CLI **headlessly** via `System.Diagnostics.Process` (`UseShellExecute=$false`, `CreateNoWindow=$true`) — **no** VS Code command-approval popup. The custom-command vector is therefore **not** gated by a dialog; the backstop is operator-trust + the layers below + a loud documentation warning.
- **Loud security warning** required in the design note, the `autopilot.host.schema.json` top-level `description`, and the skill's custom-host section: `command` runs headlessly with no approval prompt; only point it at a trusted binary; an untrusted build/test script can plant `.autopilot.host.json` for the next run (RISK-5 local-persistence vector).
- **Layers:** (1) committed `.autopilot.host.json.example` template (never read, never executed); (2) `.gitignore` the real `.autopilot.host.json` and `.autopilot.json` → never enter VCS, cannot reach a clone/container/sandbox; (3) **agent + skill absolute rule** — never read, create, or modify `.autopilot.host.json`/`.example` (launcher is sole reader); (4) **launcher hardening** — resolve `command` once to an absolute path, reuse across phases, **reject any `command`/`args` containing shell metacharacters** (`; & | $ \` < > newline`, plus cmd-significant `% ^ ( ) , !` for `.bat`/`.cmd`). "No shell" holds only for the direct-`.exe` branch; for `.bat`/`.cmd`/`.ps1` the metachar denylist is the primary control.
- **`AUTOPILOT_DISABLE_HOST=true`** disables host mode org-wide from **outside** the repo (env var, so the agent can't durably re-enable it via repo config): (a) the skill **omits Host** from the sub-menu; (b) **`launch.ps1` refuses `-Runtime host` and exits with an error** — launcher-level enforcement, net-new in this plan (the existing `AUTOPILOT_CONTAINER` is only *set* + honored by the menu/agent rule, not enforced in `launch.ps1`).
- **Residual local-persistence risk accepted & documented** in RISK-5.

### Tooling, tests, verification

- **Path edits are minimal.** Repo-root resolution stays on the existing `git rev-parse --show-toplevel` (location-independent, worktree-safe — no custom resolver added). Only **sibling bundle-file references** (devcontainer Dockerfile/context, `container-entrypoint.sh`, schema file location) change — resolve them relative to `$PSScriptRoot`.
- **Pester (first harness):** `tests/autopilot/Resolve-HostCommand.Tests.ps1` covers `Resolve-HostCommand` from the `param()`-less `plugins/autopilot/scripts/host-command.ps1` — absent→`copilot`; valid→`command`+`args`+`type`; malformed JSON→throws; empty `command`→throws; metachar in `command`/`args`→throws. Tests dot-source the helper only; launch no processes. Run: `Invoke-Pester ./tests/autopilot`.
- **Registry & dogfood integrity:** regenerate `registry.json` via `Build-Registry.ps1`; run `Sync-Dogfood.ps1` to converge `.github/`; confirm registry-wide `dest` uniqueness still holds (new dests all under `.github/skills/autopilot/`); `Test-Registry.ps1` + `validate.ps1` green.
- **Verification:** structural (files moved, manifest `files[]` complete, registry hashes match, dogfood byte-equivalent, `/ci` references the skill by path, autonomous-mode prose removed from `/ci`, agent no-read/no-write rule present as #10, `.gitignore` entries present, templates validate) **plus** PSScriptAnalyzer zero-warning on changed `.ps1` and a passing Pester run.
- **Roles:** all steps `@ai-agent`. **Execution-mode:** manual, phase scope.

# 005: Plugin Eval Harness

<!-- execution-mode: manual -->
<!-- scope: phase -->

> Implements the reserved **Plan 002 eval seams** into a working, two-tier eval harness that verifies every plugin's payload (agents, prompts, skills) keeps working as bundles change. **Tier 1 — structural/contract evals** (deterministic, no model, always-on): per-plugin Pester `*.Tests.ps1` under `plugins/<name>/evals/` assert frontmatter validity + required keys, type-aware name/slug match, referenced files/assets exist, internal markdown links resolve **from the simulated install (`dest`) base**, required body sections present, and design-note references resolve. **Tier 2 — LLM behavioral evals** (opt-in, `-IncludeLlm`, off by default, never in the CI gate): declarative `evals/llm/*.eval.json` scenarios run the artifact under test through the **Copilot CLI headless** backend **inside a disposable git clone** (push blocked, synced from source, discarded after — so the live tree is never mutated), then an **LLM-as-judge** scores the captured output against a rubric and returns a strict-JSON verdict (validated by explicit dependency-free checks). Entry point `scripts/skalary/Test-Evals.ps1` (+ `npm run eval`) discovers and runs all plugin evals and emits a gitignored `.eval-report.json` + console summary. **Writeback is report-only** — `plugin.json` / `registry.json` / receipts stay byte-static so the registry stays deterministic and the dogfood drift-check keeps passing; the reserved `evals.status` / `evalStatus` seams are left as-is. Covers all **7** existing plugins (autopilot, cdn, ci, cip, cr, dr, udn).

## Decisions

- **Two tiers, one harness.** Tier 1 (structural) is deterministic, model-free, and runs every time. Tier 2 (LLM behavioral) is opt-in behind `-IncludeLlm`, off by default, and never wired into the build/test gate. Both are discovered and orchestrated by a single entry point `scripts/skalary/Test-Evals.ps1`.
- **Runner = Pester 5.** Reuses the existing repo harness convention (`tests/skalary/*.Tests.ps1`, `Invoke-Pester`). Per-plugin structural evals live in `plugins/<name>/evals/*.Tests.ps1`. The `evals/` folder is already excluded from install (Plan 002), so eval files never ship in a plugin payload.
- **Shared assertion module.** Structural assertions are factored into `tests/evals/EvalCommon.psm1` (frontmatter parse, required-key tables per artifact type, file-exists resolver, markdown-link resolver, section-presence checker). Per-plugin `*.Tests.ps1` import this module and call its functions — DRY, one place to fix an assertion.
- **Minimal frontmatter parser — top-level scalar presence only, no external YAML module, no code-exec (DR1-#6).** The required keys we assert are **all top-level scalars** (`name`, `description`, `agent`, `user-invocable`, `disable-model-invocation`), so the parser does **not** need to descend into nested mappings or arrays. `EvalCommon`'s `Get-PluginFrontmatter` extracts the leading `---`…`---` block and captures **only column-0 `key:` lines** (`^[A-Za-z][\w-]*:`); it records each top-level key's presence and, for scalar values, the (quote-stripped) value — handling single/double-quoted scalars that contain embedded `:`/`(`/`)`/`/`. Block sequences (`handoffs:`), inline arrays (`tools: [...]`), and their nested/continuation lines are treated as **opaque presence** (the key is recorded; the value is not parsed). It never calls `Invoke-Expression`/`iex` and never dot-sources payload. A genuinely malformed block (no closing `---`, or no top-level keys) → throws and the eval fails. The parser is unit-covered against the **real** frontmatter of `cr`/`dr` (block sequences + inline arrays), `cip`/`ci` (long quoted `description` with colons/parens/slashes + `context`), and the simple prompt/agent forms. (Keeps the harness dependency-free like `validate.ps1`.)
- **Required-key tables per artifact type** (verified against current payloads):
  - **agent** (`agents/*.agent.md`): `name`, `description` required; `model`, `tools`, `agents`, `handoffs`, `argument-hint` optional.
  - **prompt** (`prompts/*.prompt.md`): `name`, `description`, `agent` required.
  - **skill** (`skills/*/SKILL.md`): `name`, `description`, `user-invocable`, `disable-model-invocation` required; `argument-hint`, `context` optional.
  - Artifact type is derived from the `dest` path suffix in `plugin.json` (`*.agent.md` → agent, `*.prompt.md` → prompt, `SKILL.md` → skill).
  - **Required scalar keys must be non-empty; duplicate-key precedence is defined (DR2-#13):** `Test-RequiredKeys` asserts each required scalar key is present **and** has a non-empty value (a bare `description:` fails); the parser uses **last-occurrence-wins** for duplicate top-level keys and this is unit-covered.
- **Per-artifact, not per-plugin, assertions.** Multi-artifact bundles (`cr` = orchestrator agent + 3 subagents + prompt + diff scripts; `dr` likewise) are iterated via `plugin.json` `files[]`; each markdown artifact is asserted on its own type. Non-markdown referenced files (e.g. `cr`'s `scripts/get-diff-*.ps1`, `cip`'s `assets/plan-template.md`) get a file-exists check only.
- **Structural checks enforced** (per artifact, all selected during interview): (1) valid frontmatter + required keys for its type; (2) **type-aware name match (DR1-#11)** — agent/prompt `name` equals the filename stem (`cdn.prompt.md`→`cdn`); skill `name` equals the **containing folder** (`skills/<name>/SKILL.md`→`<name>`); and matches the owning `plugin.json` convention; (3) referenced files exist — **narrowed extraction set (DR1-#12):** only (a) the artifact's own `plugin.json` `files[]` entries, (b) **markdown links** in the body, and (c) an explicit per-plugin known-asset list (e.g. `cip`→`assets/plan-template.md`). Arbitrary prose, command examples, and user-supplied/placeholder paths are **not** scanned (workflow skills name many future/user paths); (4) internal markdown links resolve; (5) required body sections present — see the per-type table below; (6) design-note references (`docs/design-notes/**` paths) named in a markdown link resolve.
- **Markdown/file resolution uses the simulated install (`dest`) base, repo-confined and traversal-safe (DR1-#2).** Payload links are authored relative to the **install** location (`.github/...`), **not** the source dir — e.g. `cdn.prompt.md`'s `../../docs/design-notes/.design-notes.md` is valid from `.github/prompts/` (depth 2) but not from `plugins/cdn/prompts/` (depth 3). The resolver therefore maps each artifact to its `plugin.json` `dest`, computes a **simulated `.github/<dest>`** path, resolves relative links against *that* directory, full-path-normalizes, and asserts the target stays inside the repo root and exists. Edge cases (DR1-#15): intra-page anchors (`#section`, link starts with `#`) are **skipped**; `http(s)://`/`mailto:` are **skipped** (not network-checked); trailing `#frag` is stripped before existence checks; reference-style/undefined-destination links are skipped. A relative link escaping the repo root **fails** (traversal guard, mirroring the registry stance).
- **Required body sections — explicit per-type table (DR1-#10):** skill (`SKILL.md`) → must contain ≥1 `#` heading **and** a numbered-step / `## Step`-style procedure; prompt (`*.prompt.md`) → ≥1 `#` heading **and** a procedure or imperative body (≥1 numbered/`##` section); agent (`*.agent.md`) → ≥1 `#` heading **and** a non-empty body. No "where applicable" ambiguity — each type maps to a fixed rule.
- **Schema files are documentation; validation is explicit + dependency-free (DR1-#3).** The repo has **no** JSON-Schema validator and `Test-Json -Schema` does not reliably support draft 2020-12 (Plan 003 precedent). `schemas/eval-case.schema.json` is committed as **documentation/IDE aid only**; the harness validates eval-case JSON and judge-verdict JSON with **explicit PowerShell field/type/range/`additionalProperties` checks** (required keys present, correct types, `passThreshold`/`score` in 0..1, no unknown keys), not a schema engine.
- **All 7 plugins get real structural evals — no empty `evals/` folders.** Even single-file plugins (`cdn`, `udn`) get a real `*.Tests.ps1`. This also clears the existing warn-only `Test-Registry.ps1` check ("Plugin 'X' has no evals/ folder") for every plugin.
- **Entry point `scripts/skalary/Test-Evals.ps1`.** Discovers `plugins/*/evals/**/*.Tests.ps1`, runs them via `Invoke-Pester`, aggregates results, writes `.eval-report.json` + a console summary, and exits non-zero on any failure. `-IncludeLlm` switch (default off) additionally runs the Tier-2 LLM cases. **Not** wired into `scripts/validate.ps1` or `npm test` — the build/test gate stays structural-syntax-only as today. A convenience `npm run eval` script is added.
- **Report-only writeback (reconciled).** skalary's own repo has **no** `.github/.skalary/receipts/` — `Sync-Dogfood.ps1` is a direct copy, not an install, so there is nothing to write eval status to in-repo. Therefore the harness writes results **only** to a gitignored `.eval-report.json` (+ console). `plugin.json`, `registry.json`, and the receipt `evalStatus` seam are left **static** so the registry stays deterministic and the dogfood byte-equivalence drift-check keeps passing. The `evals.status`/`evalStatus`/`lastRun` seams remain reserved (unchanged); receipt/registry writeback is **deferred** (recorded as a future enhancement, not in scope).
- **LLM tier — opt-in, off by default, out of the CI gate.** Tier 2 only runs with `-IncludeLlm` **and** when Copilot CLI auth is present; otherwise it is skipped (not failed) so structural-only runs stay green without auth. LLM cases are never part of `validate.ps1`/`npm test`.
- **Tier 2 runs in a single disposable sandbox that mirrors the live working tree — never the live tree itself (DR1-#1, DR2-#1, DR2-#3, DR3-#1, DR3-#5).** Headless `--allow-all` agents (`autopilot` commits/builds; `cr`/`dr` run git + fan out to subagents) **mutate the working tree**, which would break the report-only/byte-static invariant. So Tier 2 runs inside a throwaway sandbox: `New-EvalSandbox` clones the repo to a temp dir **once per `-IncludeLlm` run** (not per case — DR2-#3), then **replaces the clone's `plugins/` outright with the live working-tree `plugins/` (delete-then-copy, not merge — DR3-#5)** so deleted/renamed files don't linger and the overlay (including untracked `evals/llm/*.eval.json`) is the exclusive source of truth — Tier 2 judges the **current** payload under development, not committed HEAD (DR2-#1). It then runs `Sync-Dogfood.ps1` *in the sandbox* so `.github/` matches the overlaid `plugins/`. **Push is blocked while keeping `Build-Registry.ps1` working (DR3-#1):** the sandbox's `origin` **fetch** URL is set to the real GitHub URL (read from the live repo's `git remote get-url origin` before cloning — a local-path clone otherwise yields a filesystem origin that `Resolve-OriginRepository`'s `github.com[:/]owner/repo` regex rejects), and **push is disabled** via `git remote set-url --push origin DISABLED`. 3.2 asserts `Build-Registry.ps1` succeeds in the sandbox. All scenarios run in this one sandbox; a single `finally` discards it. **Honest isolation scope (DR2-#2):** the sandbox bounds *cwd*, not filesystem reach — under `--allow-all` an agent could still write to the live repo path directly; the reserved **`container` backend is the real isolation boundary** and is named as such in the design note. The earlier `Sync-Dogfood.ps1 -WhatIf` "sync-freshness drift gate" is **dropped (DR2-#4/#5)** — overlaying the live `plugins/` makes it both unnecessary and wrong (it checked live `.github/` vs `plugins/`, unrelated to what the agent executes).
- **LLM backend = Copilot CLI headless, pluggable, hard timeout with tree-kill (DR1-#9, DR2-#10).** `Invoke-EvalBackend -Backend copilot-cli` runs (inside the sandbox) `copilot -p "<scenario>" --agent <name> --no-ask-user --allow-all` for agents; for prompts/skills the artifact **body is inlined** into the scenario prompt (no `--agent`) with an **input-size ceiling note (DR2-#12)** (body+scenario+rubric must fit context; oversized → `skip` with a note). It enforces a **hard per-invocation timeout** (config-driven, default e.g. 300s) via `System.Diagnostics.Process`, killing the **entire process tree** with `$process.Kill($true)` (the CLI spawns child Node processes; plain `.Kill()` orphans them and locks the temp clone — improves on `launch-host.ps1`). The captured transcript is **copied out of the sandbox** to a stable gitignored `.eval-artifacts/<plugin>-<case>.txt` **before** disposal so `transcriptPath` in the report is live (DR2-#9). A `container` backend is reserved (backend-agnostic case/report contract) so swapping to stronger isolation later is localized. **Decision (interview):** disposable sandbox now; keep all artifact types incl. `cr`/`dr`, accepting the CLI-fidelity caveat below.
- **Transcript capture is spiked before the backend is built (DR1-#5).** autopilot's existing mechanism captures live streaming + exit codes + git state, **not** clean assistant prose. The exact way to obtain the assistant's text output from `copilot -p` (e.g. `--share=<file>` location/format, completion detection, separating tool/stderr noise from assistant text, max size, timeout, cleanup) is **unproven** and is validated by an explicit spike step (3.0) against the real CLI; the backend contract is fixed only after the spike confirms it.
- **`cr`/`dr` CLI-fidelity caveat (DR1-#7, accepted).** `cr`/`dr` are VS-Code-hosted orchestrators (qualified `Model Name (vendor)` form) that fan out to three subagents; run via `copilot --agent` they may not reproduce subagent fan-out or model-vendor resolution faithfully. The user chose to keep them in Tier 2 anyway; their LLM rubrics target **observable orchestrator behaviour** (e.g. injection-guarding diff content, emitting a structured findings list) rather than exact multi-model consensus, and the fidelity limitation is recorded as RISK-11 and documented in the design note.
- **Declarative LLM case format `plugins/<name>/evals/llm/*.eval.json`:** `{ "artifact": "<dest-relative path>", "scenario": "<user prompt>", "rubric": ["criterion", …], "passThreshold": <0..1> }`, documented by `schemas/eval-case.schema.json` and enforced by the harness's **explicit field/type checks** (DR1-#3). One shared loader interprets every case generically; no per-case PowerShell. **Body-inlining is an explicit non-goal note (DR1-#13):** inlining a prompt/skill body into `-p` approximates the artifact's content but does **not** exercise real slash-command dispatch or `disable-model-invocation`; results are interpreted accordingly.
- **LLM-as-judge, structured-JSON verdict, temperature 0, injection-guarded (boundary hardened — DR2-#11).** After the backend captures the artifact's output, a second Copilot CLI call (the judge) receives the rubric + the captured output and must return **strict JSON** `{ "pass": bool, "score": 0..1, "rationale": "…" }`. The harness validates the verdict with explicit field/type/range checks and **rejects non-JSON** (fails the case loudly rather than guessing). The captured output is wrapped in **per-call randomized (GUID-suffixed) boundary markers** `<<<UNTRUSTED_OUTPUT_START:{guid}>>>` / `<<<UNTRUSTED_OUTPUT_END:{guid}>>>` with quad-tick fencing before being placed in the judge prompt — exactly the `cr`/`dr` injection-guard pattern — and the captured output is scanned/neutralized for any literal boundary token before wrapping so it cannot break out of the fence. The judge is told to treat marked content as data only. `pass` requires `score >= passThreshold`.
- **Auth + model config (skip-not-error — DR2-#7).** LLM evals use **Copilot CLI ambient auth** (same as autopilot — no API key handled by the harness). Judge model slug, default temperature, default `passThreshold`, and the backend **timeout** live in a **gitignored** `.eval.config.json`; a committed `.eval.config.json.example` documents the shape. **No model identity is hardcoded** in committed files. **Missing-auth/config is always a `skip`, never an error (DR2-#7):** when `-IncludeLlm` is set but `copilot` is absent/unauthenticated, or a required config value (e.g. `judgeModel`) is unresolved, the affected Tier-2 cases are recorded as `skip` with an actionable note and the run stays **green** if structural evals pass. `error`/non-zero is reserved for backend/judge failures **after** preflight succeeds. Optional fields fall back to `.example` defaults.
- **Expanded report schema (DR1-#14), one canonical shape (DR2-#6).** **Every** `.eval-report.json` entry (structural and LLM) uses the same shape: `{ plugin, case, artifact, tier, outcome (pass|fail|skip|error), score?, threshold?, message?, transcriptPath? }`. Structural entries set `case`=test file/name, `artifact`=the asserted file, `message`=failure text; LLM entries add `score`/`threshold`/`transcriptPath`. Step 1.2 emits this canonical shape from the start so the LLM merge (3.5) does not reconcile two schemas.
- **LLM coverage.** At least **one** `llm/*.eval.json` scenario per plugin (7 minimum). Complex skills/agents get **multiple** scenarios: `cip`, `ci`, `cr`, `dr` (each exercises several distinct behaviours). Simple single-purpose artifacts (`cdn`, `udn`, `autopilot`) get one each.
- **Security.** Plugin payload **and** captured model output are treated as untrusted **data** — never executed, dot-sourced, or interpolated into a shell/git command. The frontmatter/link parsers are pure string ops. The judge prompt wraps untrusted output in markers and the verdict is strict-JSON-validated. Headless `--allow-all` runs trusted, committed scenarios in the repo; the no-approval risk is documented in the design note, and the pluggable backend allows moving to container isolation if a riskier scenario is ever added. PSScriptAnalyzer zero-warning on all changed `.ps1`/`.psm1`; every script declares `#requires -Version 7.0`.
- **Design notes.** New `docs/design-notes/architecture/plugin-evals.design.md` (two-tier model, file layout, backend seam, judge contract, report-only writeback, security). Update `plugin-registry.design.md` ("Evals (Future) Contract" → "Evals Contract", now implemented; note report-only writeback and that seams remain reserved), `copilot-customizations.design.md` (mention the harness + `npm run eval`), and the `.design-notes.md` index.
- **Roles:** all steps `@ai-agent`. **Execution-mode:** manual, phase scope.

## Risks

|----|------|------------|--------|------------|-------|

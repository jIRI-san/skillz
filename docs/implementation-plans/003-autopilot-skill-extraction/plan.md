# 003: Autopilot Skill Extraction

<!-- execution-mode: manual -->
<!-- scope: phase -->

> Extracts the three **autonomous** execution modes (host / container / sandbox) out of the `/ci` skill's inline Step 1 logic into a dedicated, cross-referenceable `autopilot` **skill** (`.github/skills/autopilot/SKILL.md`). `/ci`'s mode menu collapses to **Approve / Autopilot / Autonomous**; choosing *Autonomous* hands off to the new skill, which owns the host/container/sandbox sub-menu, the branch follow-up question, and the `launch.ps1` invocations. Adds a **custom host launch command** seam (operator-provisioned, gitignored `.autopilot.host.json`) and **wires it for real into the now-implemented `launch-host.ps1`**. The in-editor **Approve** and **Autopilot** modes stay inline in `/ci`. No changes to container/sandbox runtime behavior, to the `.autopilot.json` shape, or to Plan 002.

## Reconciliation with Plan 001 (delivered)

Plan 001 is **complete** and shipped **all three runtimes** (host/container/sandbox) — its *plan text* differed from what was built. This plan is reconciled to the **delivered code**, verified by reading the scripts:

- **Real launcher signature** (from `scripts/autopilot/launch.ps1`): `launch.ps1 -PlanSlug <slug> -Mode whole-plan|next-phase -Runtime host|container|sandbox [-Branch <branch>]`. This **matches the original `/ci` text** — the earlier DR-driven "reconcile to `-Mode host|container`" was based on stale plan text and is **reverted**.
- **Sandbox is fully implemented** (`scripts/autopilot/launch-sandbox.ps1`) — **not** deferred. Treated as an available mode.
- **`-Branch` exists** on `launch.ps1` and `launch-host.ps1` — not forward-spec.
- **Host per-phase invocation** lives in `launch-host.ps1` `Invoke-CopilotPhase`, which hardcodes `Get-Command copilot` + args `-p "<prompt>" --agent autopilot --no-ask-user --allow-all --share=./<transcript>`. This is the **seam** the custom host command plugs into — a **real, implementable edit now** (no longer forward-spec).
- **Existing schema** is JSON Schema **draft 2020-12** (`schemas/autopilot.schema.json`); the new host schema matches that draft.
- **Lint/format tooling exists**: `PSScriptAnalyzerSettings.psd1` + `.editorconfig` at repo root.

## Decisions

- **Extraction boundary.** Only the three autonomous modes (Host / Container / Sandbox) move into the new skill. The two in-editor modes (Approve, Autopilot) remain inline in `/ci` Step 1 because they run inside the editor agent loop, not via `launch.ps1`.
- **`/ci` menu collapses to 3 options:** Approve · Autopilot · **Autonomous**. The prior five-way menu (which listed Host/Container/Sandbox directly) is replaced. Choosing *Autonomous* makes `/ci` load `.github/skills/autopilot/SKILL.md` and follow it; the skill presents the Host/Container/Sandbox sub-menu.
- **Skill identity:** `.github/skills/autopilot/SKILL.md`, name `autopilot`. Same name as the existing `autopilot` **agent** (`.github/agents/autopilot.agent.md`) — VS Code distinguishes by type (skill vs agent). The agent remains the per-phase executor invoked by `launch.ps1`; the skill is the in-editor mode-selection + orchestration entry point. Both share the "autopilot" identity intentionally.
- **Skill is internal-only; consumed as a textual include, not a skill invocation (DR1-#11).** Frontmatter: `user-invocable: false`, `disable-model-invocation: true`. With both invocation paths disabled the file is never invoked *as a skill* — `/ci` **reads it by path** and follows its steps inline within `/ci`'s existing context. No `context: fork` (inert here; would imply skill-invocation semantics). No new `/autopilot` slash command. The new design note documents this include-vs-invoke distinction explicitly.
- **`AUTOPILOT_CONTAINER=true` suppresses *Autonomous*** in `/ci` (already inside an autonomous container → only Approve/Autopilot offered). This guard, previously applied to the Host/Container/Sandbox entries, now applies to the single *Autonomous* entry.
- **`launch.ps1` invocations preserved verbatim against the delivered signature.** The skill issues the real commands: `scripts/autopilot/launch.ps1 -PlanSlug <slug> -Mode whole-plan -Runtime host|container|sandbox [-Branch <chosen-branch>]`. No argument changes for any runtime. Zero behavior change to autonomous runs. (Reverts the earlier mistaken reconciliation.)
- **Sandbox is an available mode.** The skill lists Host/Container/Sandbox and invokes `-Runtime sandbox` for real; `launch.ps1` already runs a Windows-Sandbox pre-flight and dispatches to `launch-sandbox.ps1`.
- **Branch follow-up preserved.** Container and Sandbox modes keep the "Start from which branch? (Current branch / main)" follow-up; the chosen value is passed via the real `-Branch` parameter. Host mode uses its own `feature/<slug>` worktree (no follow-up).
- **Custom host launch command (new) — implemented in `launch-host.ps1`:** Host mode may run a custom Copilot CLI executable (e.g. a company wrapper that injects MCP servers + internal skills) instead of vanilla `copilot`.
  - Config lives in a **separate, operator-provisioned, gitignored** file `.autopilot.host.json` at repo root — **not** in `.autopilot.json` (which the autopilot agent may edit for `build`/`test`).
  - Fields: `command` (string — executable name or full path, e.g. `mycorp-copilot`), `args` (string[] — static extra args added to each per-phase invocation, e.g. `--mcp-config` flags, alongside the standard `--agent autopilot --no-ask-user --allow-all --share` flags). No `displayName` field (the Host menu label is static — see below).
  - **Implementation — resolution vs. invocation are separate concerns (DR2-#2):** the real `Invoke-CopilotPhase` does a **single** `Get-Command copilot` to resolve a path, then branches at **invocation time** into three cases — `.bat`→`cmd.exe /c`, `.ps1`→`powershell.exe -File`, else→direct exe — each building `$psi.Arguments` (a **string**, not `ArgumentList`). There is **no `.cmd` branch today** (a `.cmd` target falls through to direct exec and fails under `UseShellExecute=$false` — and npm global shims on Windows are typically `*.cmd`).
    - Extract a `param()`-less helper `scripts/autopilot/host-command.ps1` exporting `Resolve-HostCommand` (DR2-#1 — see Pester decision). It reads `.autopilot.host.json` once, validates, resolves `command` to an absolute path, classifies its **type** (`exe`/`bat`/`cmd`/`ps1` by extension), and returns `@{ Path; Type; ExtraArgs }`. `launch-host.ps1` dot-sources this helper and calls it once before the phase loop, replacing the hardcoded `Get-Command copilot`.
    - `Invoke-CopilotPhase` keeps its existing **per-type invocation branching** driven by `Type`, **adds an explicit `.cmd`→`cmd.exe /c` branch** (the real gap), and appends `ExtraArgs` tokens into the per-branch argument construction. No wholesale `ArgumentList` migration (the existing code uses `$psi.Arguments` strings; that is preserved).
  - **State handling (DR1-#6):** absent → default `copilot` with no extra args; **present-but-invalid** (malformed JSON, schema violation, empty `command`, or shell metacharacter in `command`/`args`) → **fail loudly before any phase starts** with an actionable error — never silent-fallback to `copilot` (mirrors Plan 001's no-silent-fallback stance).
  - Validated by `schemas/autopilot.host.schema.json` (draft 2020-12).
  - **Host-only.** `Resolve-HostCommand` is dot-sourced only by `launch-host.ps1`; container (`launch-container.ps1`) and sandbox (`launch-sandbox.ps1`) never read the file (they clone from remote and run vanilla CLI by design; the file is gitignored so it never reaches them anyway).
- **Static Host menu label (DR1-#1).** The skill shows a fixed "Host autopilot" label and **does not read `.autopilot.host.json`** to render a dynamic name. Reading it would violate the "launcher is sole reader" invariant and the agent absolute rule (which forbids *reading* the file). The `displayName` field is therefore dropped entirely.
- **No interactive approval for the host launcher (honesty, DR1-#4).** `launch-host.ps1` runs the Copilot CLI **headlessly** via `System.Diagnostics.Process` (`UseShellExecute=$false`, `CreateNoWindow=$true`) — it does **not** route through VS Code's command-approval prompt. The executable named in `command` is launched directly with no popup. The custom-command vector is therefore **not** gated by an approval dialog; the backstop is operator-trust + a loud documentation warning (below), not an interactive gate.
- **Loud security warning required.** The new design note, `schemas/autopilot.host.schema.json` top-level `description`, and the skill's custom-host section all carry an explicit warning: `command` is executed headlessly with no approval prompt; only point it at a trusted binary; an untrusted build/test script can plant `.autopilot.host.json` for the next run (RISK-6 local-persistence vector).
- **Host mode is disable-able org-wide via `AUTOPILOT_DISABLE_HOST=true` (env var).** Mirrors the env-var **convention** of `AUTOPILOT_CONTAINER` — but note `AUTOPILOT_CONTAINER` is **not** enforced inside `launch.ps1` (it is only *set* by `prepare-env-file.ps1` and *honored* by agent Absolute Rule #8 + the `/ci` menu). The launcher-level enforcement here is therefore **net-new in this plan**, not a copy of an existing block (DR2-#4). When set: (a) the autopilot skill **omits Host** from the autonomous sub-menu (Container/Sandbox only); (b) **`launch.ps1` refuses `-Runtime host` (or `runtime:host` from config) and exits with an error** — launcher-level enforcement, not just an advisory menu omission. Env var (not `.autopilot.json`) because it lives **outside the repo**, so the autopilot agent cannot durably re-enable host mode by editing repo config.
- **Custom-command security model — defense in depth (independent layers):**
  1. **Committed template** `.autopilot.host.json.example` documents the shape; never read by the launcher, never executed.
  2. **`.gitignore` the real `.autopilot.host.json`** → never enters VCS, so it cannot reach a clone/container/sandbox and cannot be injected via a PR/diff. The custom command exists only on the trusted operator's host. (Honest scope: gitignore is a VCS filter, **not** a filesystem write-guard.)
  3. **Absolute agent rule (read *and* write):** the autopilot agent and the skill **never** read, create, or modify `.autopilot.host.json`. The host launcher is the **sole reader**. This stops in-run self-tampering (gitignore cannot) and keeps the menu-label invariant (DR1-#1) intact. Added to `.github/agents/autopilot.agent.md` Absolute Rules and stated consistently in the skill (no write-only carve-out).
  4. **Launcher hardening (implemented):** `Resolve-HostCommand` resolves `command` **once** at launch to an absolute path, reuses it across all phases, and **rejects any `command`/`args` value containing shell metacharacters** (`; & | $ \` < > newline`, plus cmd-significant `% ^ ( ) , !` for the `.bat`/`.cmd` path) — mirroring Plan 001's build/test command policy. **Honest "no shell" scope (DR2-#3):** a direct `.exe` target runs with no shell; a `.bat`/`.cmd` target runs via `cmd.exe` and a `.ps1` via `powershell.exe`, where the **expanded metachar denylist is the primary control** (.NET argument strings are not safely escaped for `cmd.exe`). The denylist plus the operator-trust + gitignore + agent-rule layers are the defense for shell-wrapped custom commands.
- **Pester coverage (new test harness, DR2-#1).** This is the repo's first Pester suite: `tests/autopilot/Resolve-HostCommand.Tests.ps1` covers the pure config logic in the **`param()`-less helper** `scripts/autopilot/host-command.ps1` — absent → `copilot`; valid → `command`+`args`+`type`; malformed JSON → throws; empty `command` → throws; metacharacter in `command`/`args` → throws. The helper is extracted into its own file precisely because `launch-host.ps1` declares four `[Parameter(Mandatory)]` params that would prompt on dot-source — a body-level `if` guard cannot prevent param binding. Tests dot-source `host-command.ps1` only, launching no processes.
- **No edits to Plan 002.** Plan 002's plugin inventory currently models an `autopilot` plugin = the agent file only; adding an `autopilot` skill changes that inventory. This plan does **not** touch Plan 002; the drift is recorded as RISK-2.
- **Skill's owning plugin = `ci` (DR1-#3).** When Plan 002 packages plugins, the new `.github/skills/autopilot/SKILL.md` belongs to the **`ci` plugin** (its sole consumer; `ci` already depends on `autopilot`), not the `autopilot` plugin (which stays agent-file-only to preserve `.github/`-confinement). Embedded durably in the new design note **and** the `copilot-customizations.design.md` File Inventory.
- **Plan-number collision note (DR1-#9).** Plan 002's prose references a *future Plan 003 = eval harness*, but this folder (`003-autopilot-skill-extraction`) took that number. Recommendation (not actioned here): renumber the eval harness to **Plan 004**.
- **Design notes:** update `copilot-customizations.design.md` (file inventory + `ci` flow), `autopilot-execution.design.md` (architecture diagram: `/ci` → autopilot skill → `launch.ps1`; custom host command section), and `.design-notes.md` index; add new `architecture/autopilot-skill.design.md` documenting the skill's responsibility, the `/ci` handoff contract, and the `.autopilot.host.json` config.
- **Verification:** structural for markdown/JSON (files exist, valid frontmatter, schema + `.example` validate, `/ci` references the skill by path, autonomous-mode prose removed from `/ci`, agent no-read/no-write rule present, `.gitignore` entry present) **plus** PSScriptAnalyzer zero-warning gate on changed `.ps1` files and a passing Pester run for `Resolve-HostCommand`.
- **Roles:** all steps `@ai-agent`. **Execution-mode:** manual, phase scope.

## Phase 2: Implement custom host command in the launcher + Pester
<!-- worktree: (recorded by /ci when worktree is created) -->

- [ ] 2.1 Extract a **`param()`-less helper** `scripts/autopilot/host-command.ps1` exporting `Resolve-HostCommand` (separate file because `launch-host.ps1`'s four `[Parameter(Mandatory)]` params bind before any body guard runs, so it cannot be dot-sourced without prompting — DR2-#1). `Resolve-HostCommand`: (a) reads `.autopilot.host.json` from the repo root if present; (b) **absent → returns `@{ Path = (Get-Command copilot).Source; Type = 'exe'; ExtraArgs = @() }`** (strict no-op vs today's behavior); (c) present → parses JSON, validates `command` is a non-empty string and `args` (if any) is a string[], **rejects any `command`/`args` value containing shell metacharacters** (`; & | $ \` < > newline` + cmd-significant `% ^ ( ) , !`), resolves `command` to an absolute path via `Get-Command`/full-path, **classifies its type** (`exe`/`bat`/`cmd`/`ps1` by extension), and returns `@{ Path; Type; ExtraArgs }`; (d) present-but-invalid → **throws a loud, actionable error** (never silent-fallback). Then edit `scripts/autopilot/launch-host.ps1` to dot-source the helper, call `Resolve-HostCommand` **once** before the phase loop, and thread `Path`/`Type`/`ExtraArgs` into `Invoke-CopilotPhase`, replacing its hardcoded `Get-Command copilot` resolution. In `Invoke-CopilotPhase`, keep the existing per-type **invocation** branching (`.bat`→`cmd.exe /c`, `.ps1`→`powershell.exe -File`, direct-exe), **add the missing `.cmd`→`cmd.exe /c` branch** (DR2-#2), and append `ExtraArgs` tokens into each branch's `$psi.Arguments` construction (the existing string approach is preserved; no wholesale `ArgumentList` migration).

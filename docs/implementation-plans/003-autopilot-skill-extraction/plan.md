# 003: Autopilot Skill Extraction

<!-- execution-mode: manual -->
<!-- scope: phase -->

> Extracts the three **autonomous** execution modes (host / container / sandbox) out of the `/ci` skill's inline Step 1 logic into a dedicated, cross-referenceable `autopilot` **skill** (`.github/skills/autopilot/SKILL.md`). `/ci`'s mode menu collapses to **Approve / Autopilot / Autonomous**; choosing *Autonomous* hands off to the new skill, which owns the host/container/sandbox sub-menu, the branch follow-up question, and the `launch.ps1` invocations. Adds a **custom host launch command** seam (operator-provisioned, gitignored `.autopilot.host.json`) so host mode can run a company-internal Copilot CLI wrapper. The in-editor **Approve** and **Autopilot** modes stay inline in `/ci`. No changes to container/sandbox runtime behavior, to `.autopilot.json`, or to Plan 001 / Plan 002.

## Decisions

- **Extraction boundary.** Only the three autonomous modes (Host / Container / Sandbox) move into the new skill. The two in-editor modes (Approve, Autopilot) remain inline in `/ci` Step 1 because they run inside the editor agent loop, not via `launch.ps1`.
- **`/ci` menu collapses to 3 options:** Approve · Autopilot · **Autonomous**. The prior five-way menu (which listed Host/Container/Sandbox directly) is replaced. Choosing *Autonomous* makes `/ci` load `.github/skills/autopilot/SKILL.md` and follow it; the skill presents the Host/Container/Sandbox sub-menu.
- **Skill identity:** `.github/skills/autopilot/SKILL.md`, name `autopilot`. Same name as the existing `autopilot` **agent** (`.github/agents/autopilot.agent.md`) — VS Code distinguishes by type (skill vs agent). The agent remains the per-phase executor invoked by `launch.ps1`; the skill is the in-editor mode-selection + orchestration entry point. Both share the "autopilot" identity intentionally.
- **Skill is internal-only; consumed as a textual include, not a skill invocation (DR1-#11).** Frontmatter: `user-invocable: false`, `disable-model-invocation: true`. With both invocation paths disabled the file is never invoked *as a skill* — `/ci` **reads it by path** and follows its steps inline within `/ci`'s existing context. No `context: fork` (inert here; would imply skill-invocation semantics). No new `/autopilot` slash command. The new design note documents this include-vs-invoke distinction explicitly.
- **`AUTOPILOT_CONTAINER=true` suppresses *Autonomous*** in `/ci` (already inside an autonomous container → only Approve/Autopilot offered). This guard, previously applied to the Host/Container/Sandbox entries, now applies to the single *Autonomous* entry.
- **`launch.ps1` invocations reconciled to Plan 001's real contract (DR1-#2), not copied verbatim.** Plan 001 — the owner of `launch.ps1` — specs `scripts/autopilot/launch.ps1 -Mode host|container`; the executed plan is `planPath` from `.autopilot.json` (no `-PlanSlug`/`-Runtime`/`-Mode whole-plan`/`-Branch` exist). The live `/ci` text uses a **fictional** signature; copying it verbatim would entrench a contract that breaks when the launcher lands. The skill therefore: (a) ensures `.autopilot.json` `planPath` points at the selected plan before launching; (b) invokes `launch.ps1 -Mode host` or `launch.ps1 -Mode container`. REQ-3 verifies **semantic match to Plan 001's documented signature**, not byte-identity against old `/ci`.
- **Sandbox is forward-spec / unavailable (DR1-#2).** Plan 001 explicitly **defers sandbox** (out of scope, documented future). The skill lists Sandbox as a mode but marks it **unavailable until the sandbox orchestrator is built**, with a clear message; it does not invoke a non-existent launcher. Recorded in RISK-1.
- **Branch selection is forward-spec.** Plan 001's launcher has **no** `-Branch` parameter (host uses a `feature-<slug>` worktree; container clones from remote). The skill keeps the container/sandbox "Start from which branch? (Current / main)" follow-up but records it as a **forward-spec extension Plan 001 must honor** (e.g. a `-Branch` param or a config field); until then the skill notes the chosen branch is informational. Recorded in RISK-1.
- **Custom host launch command (new):** Host mode may run a custom Copilot CLI executable (e.g. a company wrapper that injects MCP servers + internal skills) instead of vanilla `copilot`.
  - Config lives in a **separate, operator-provisioned, gitignored** file `.autopilot.host.json` at repo root — **not** in `.autopilot.json` (which the autopilot agent may edit).
  - Fields: `command` (string — executable name or full path, e.g. `mycorp-copilot`), `args` (string[] — static extra args prepended to each per-phase invocation, e.g. MCP-config flags). **No `displayName` field** — see the displayName decision below; the Host menu label is static.
  - **State handling (DR1-#6):** absent → default `copilot` with no extra args; **present-but-invalid** (malformed JSON, schema violation, empty `command`, or shell metacharacter in `command`/`args`) → **fail loudly before any phase starts** with an actionable error — never silent-fallback to `copilot` (mirrors Plan 001 DR1-#10 no-silent-fallback).
  - Validated by `schemas/autopilot.host.schema.json`.
  - **Host-only.** Container and sandbox modes ignore the file entirely (they clone from remote and run vanilla CLI by design; the file is gitignored so it never reaches them anyway).
- **Static Host menu label (DR1-#1).** The skill shows a fixed "Host autopilot" label and **does not read `.autopilot.host.json`** to render a dynamic name. Reading it would violate the "launcher is sole reader" invariant and the agent absolute rule (which forbids *reading* the file). The `displayName` field is therefore dropped entirely.
- **No interactive approval for the host launcher (honesty, DR1-#4).** The host autopilot launcher runs the Copilot CLI **headlessly** via `launch.ps1` → `System.Diagnostics.Process` (Plan 001 step 3.2) — it does **not** route through VS Code's command-approval prompt. The executable named in `command` is launched directly with no popup. The custom-command vector is therefore **not** gated by an approval dialog; the backstop is operator-trust + a loud documentation warning (below), not an interactive gate. The plan documents this accurately rather than implying a popup that never fires.
- **Loud security warning required.** The new design note, `schemas/autopilot.host.schema.json` top-level `description`, and the skill's custom-host section all carry an explicit warning: `command` is executed headlessly with no approval prompt; only point it at a trusted binary; an untrusted build/test script can plant `.autopilot.host.json` for the next run (RISK-6 local-persistence vector).
- **Host mode is disable-able org-wide via `AUTOPILOT_DISABLE_HOST=true` (env var).** Mirrors the existing `AUTOPILOT_CONTAINER` pattern. When set, the autopilot skill **omits Host** from the autonomous sub-menu (Container/Sandbox only). Env var (not `.autopilot.json`) because it lives **outside the repo**, so the autopilot agent cannot durably re-enable host mode by editing repo config. Repo owners who never want headless host execution set it once in their environment.
- **Custom-command security model — defense in depth (all four layers independent):**
  1. **Committed template** `.autopilot.host.json.example` documents the shape; never read by the launcher, never executed.
  2. **`.gitignore` the real `.autopilot.host.json`** → never enters VCS, so it cannot reach a clone/container/sandbox and cannot be injected via a PR/diff. The custom command exists only on the trusted operator's host. (Honest scope: gitignore is a VCS filter, **not** a filesystem write-guard.)
  3. **Absolute agent rule (read *and* write):** the autopilot agent and the skill **never** read, create, or modify `.autopilot.host.json`. The host launcher is the **sole reader**. This stops in-run self-tampering (gitignore cannot) and keeps the menu-label invariant (DR1-#1) intact. Added to `.github/agents/autopilot.agent.md` Absolute Rules and stated consistently in the skill (no write-only carve-out).
  4. **Launcher hardening (forward-spec):** the host orchestrator resolves `command` **once** at launch via full-path / `Get-Command`, reuses the resolved path across all phases, executes as an **argv array with no shell** (`& $exe @args`), and **rejects any `command`/`args` value containing shell metacharacters** (`; & | $ \` < > newline`) — mirroring Plan 001's build/test command policy (DR1-#1).
- **Forward-spec caveat (cross-plan).** Plan 001 owns `scripts/autopilot/launch.ps1`, `launch-host.ps1`, and `schemas/autopilot.schema.json`, and is only partially implemented (step 1.1 done). The host-launcher pass-through for `.autopilot.host.json` (resolving `command`/`args`, hardening) is written here as the **contract Plan 001 must honor**, plus the schema + template + gitignore + agent rule (all owned by this plan). If `launch-host.ps1` does not yet exist when this plan runs, this plan delivers the spec + schema + template + agent rule + skill surface; wiring into live launcher code is gated on Plan 001 reaching its host-orchestrator step. Recorded as RISK-1.
- **No edits to Plan 001 or Plan 002.** Plan 002's plugin inventory currently models an `autopilot` plugin = the agent file only; adding an `autopilot` skill changes that inventory. This plan does **not** touch Plan 002; the drift is recorded as RISK-2 for Plan 002 to reconcile when implemented.
- **Skill's owning plugin = `ci` (DR1-#3).** When Plan 002 packages plugins, the new `.github/skills/autopilot/SKILL.md` belongs to the **`ci` plugin** (its sole consumer; `ci` already depends on `autopilot`), not the `autopilot` plugin (which stays agent-file-only to preserve `.github/`-confinement). This decision is embedded durably in the new design note **and** the `copilot-customizations.design.md` File Inventory so Plan 002's migration step reads it directly rather than relying on recall.
- **Plan-number collision note (DR1-#9).** Plan 002's prose references a *future Plan 003 = eval harness*, but this folder (`003-autopilot-skill-extraction`) took that number. Recommendation (not actioned here): renumber the eval harness to **Plan 004**. Recorded so Plan 002's author corrects the stale reference; this plan does not edit Plan 002.
- **Design notes:** update `copilot-customizations.design.md` (file inventory + `ci` flow), `autopilot-execution.design.md` (architecture diagram: `/ci` → autopilot skill → `launch.ps1`; custom host command section), and `.design-notes.md` index; add new `architecture/autopilot-skill.design.md` documenting the skill's responsibility, the `/ci` handoff contract, and the `.autopilot.host.json` config.
- **Verification is structural** (repo has no test harness for markdown customizations): files exist with valid frontmatter, schema + `.example` schema-validate (`Test-Json -SchemaFile`), `/ci` references the skill by path, the autonomous-mode prose is removed from `/ci` (no duplication), the skill's launcher invocations semantically match Plan 001's `-Mode host|container` contract, the agent no-read/no-write rule is present, `.gitignore` contains the entry.
- **Roles:** all steps `@ai-agent`. **Execution-mode:** manual, phase scope.

> Hardens the `cip`/`ci`/**autopilot** workflow's **enforcement + execution spine**, designed to run end-to-end under autopilot (the primary mode). Scope after the round-2 simplicity split (the workflow-memory **ledger + learning capture/harvest** moved to sibling plan **007**, which depends on this one): **(a)** cut bloat — make the oversized `cip`/`ci` `SKILL.md` thin orchestrators that read bundled `assets/*.md` on demand, with a preservation checklist + inline non-negotiables so no guarantee is lost, plus a state anchor; **(c)** harden the interview (no-TBD blocker, mandatory typed evidence per REQ, pre-draft gate); **(d)** a deterministic, **pre-approvable** PowerShell validator `Test-Plan.ps1` (cross-ref integrity, ID hygiene, dependency resolution, phase-budget, evidence integrity) with an explicit `-Stage` so a single stateless gate invocation behaves correctly, folded into the real `validate.ps1` gate; **(e)** make plans fit autopilot per-phase budgets and run the verification loop autonomously, escalating to a human **only when the run determines a decision is needed**; **(f)** make every requirement **provable** via machine-checkable typed markers (`test:` / `file:` / `review:`) embedded in Acceptance Criteria, verified at crosscheck and recorded in `evidence.md`, gating completion. **Cross-cutting principle:** all `cip`/`ci`/autopilot validation runs through pre-approvable `.ps1` scripts — no inline/ad-hoc validation logic. Deep `cr`/`dr` change-extraction redesign is deferred to plan 007-review (a separate follow-up); only lightweight review wins are here.

## Decisions

### Split (round 2 simplicity gate)

- **Two cohesive plans.** Round-2 DR (all 3 models, High) flagged 28 REQ/32KB as two deliverables whose coupling caused the bootstrapping Criticals. **006 = enforcement spine** (thin orchestrators + `Test-Plan.ps1` validator + evidence + autopilot-loop coherence). **007 = workflow-memory ledger + capture/harvest** (per-category ledger, `cr-log.md`/`learnings.md`, finalization harvest), built on top of 006's assets + finalization step. 007 is created as a sibling plan and sequenced after 006.

### Validation is always a pre-approvable script (cross-cutting)

- **No inline validation.** Every validation `cip`/`ci`/autopilot performs (plan integrity, evidence checks, drift checks) runs through a committed, pre-approvable `.ps1` the user can allowlist — never ad-hoc regex/logic improvised in chat or interpolated into a shell. Existing inline checks (e.g. the cross-reference one-liners used while authoring) are **extracted into `Test-Plan.ps1`**. Orchestrator instructions invoke scripts; they never embody validation logic themselves.

### `Test-Plan.ps1` — staged, deterministic, security-hardened

- **`-Stage Draft|PhaseCrosscheck|PlanCrosscheck`.** A single stateless gate run (`validate.ps1`/per-step) cannot know draft vs crosscheck, so the stage is an explicit parameter (round-2 Critical #1). **Draft** stage: structural integrity blocks, but `test:`/`file:` *target resolution* is warn-only (targets legitimately land in later steps). **Crosscheck/Plan** stages: target resolution blocks. The repo-wide `validate.ps1` invocation runs `-Stage Draft` (warn on unresolved targets) so 006 never self-blocks on later-phase artifacts.
- **Integrity scope vs opt-in.** Cross-ref + ID hygiene + dependency checks always run, but are **blocking only for plans carrying `<!-- evidence: required -->`**; legacy plans (003/004/005) without the marker are **warn-only** (round-2 High #6). Legacy step-ID grammar is suffix-tolerant: `\d+\.\d+[a-z]?` (verified — plan 004 uses `3.2a`). Legacy plans are added as passing fixtures.
- **Closed `file:` assertion vocabulary + grammar.** `file:<path>#<assertion>` where `<assertion>` ∈ `exists` · `contains:<regex>` · `count>=<N>` (file: line/match count) · `dircount>=<N>` (directory entry count, distinct verb — round-2 #9). Everything after the first `:` in `contains:` is the literal regex to end-of-token (round-2 #9). Free-form anchors are rejected.
- **`file:` evaluator security** (round-2 High #7 + round-3 #4): repo-root confinement that **canonicalizes paths (resolving symlinks) before** confining — not lexical `..` stripping, so a symlinked subdir cannot escape the root; the `dircount` directory walk **does not follow symlinks** (cycle-safe); cap file size and directory-walk breadth; `contains:` runs under both a per-match regex timeout **and a global per-file evaluation budget** (a large in-cap file can't incur the timeout thousands of times). Folded into REQ-14/REQ-25's `Test-Plan.Security.NoCodeExec` + `FileConfinement` tests.
- **`file:` evaluator is a reusable callable, not just whole-plan** (round-3 #2). The path-confined assertion evaluator is extracted into a dot-sourceable helper (`scripts/skalary/PlanEvidence.psm1`, or a `-VerifyEvidence -PlanPath` parameter set on `Test-Plan.ps1`) so `ci`/autopilot crosscheck verification calls a **real pre-approvable script** to evaluate one `file:<path>#<assertion>` → pass/fail, honoring REQ-9 (no in-chat evaluation). Both `Test-Plan.ps1` and the verify entrypoint import it.
- **Parser** skips fenced code blocks (example IDs/markers inside ```` ``` ```` ignored — round-2 #14) and normalizes table cells. Pure string ops, plan text untrusted: no `Invoke-Expression`/dot-source/shell interpolation. `#requires -Version 7.0`, PSScriptAnalyzer-clean.
- **Folded into the real gate.** `scripts/validate.ps1` runs `Test-Plan.ps1 -Stage Draft` over `docs/implementation-plans/**/plan.md` so integrity is deterministically enforced in the same gate autopilot runs as build/test. `npm run validate-plan` is the focused single-plan entry point. Orchestrator instructions are **best-effort prompts**; hard enforcement lives in the script + gate.

### Provable evidence — machine-checkable only

- **Typed markers in Acceptance Criteria** (no separate column): `test:<TestId>` (named Pester test, must exist + pass), `file:<path>#<assertion>` (closed vocab above), `review:cr|dr` (finding-class absent). `cmd:`/`manual:` removed (round-1 — untrusted exec / can't run under autopilot). All three remaining types are verifiable inside autopilot.
- **`test:unit` Pester gate (host-aware).** `npm run test:unit` runs `Invoke-Pester` over `tests/skalary/**` (existing `Skalary.Tests.ps1` + new `Test-Plan.Tests.ps1`) — gives `test:` evidence a real executor (the structural `validate.ps1` stays dependency-free). **Pester is installed in the autopilot container** via a pinned Dockerfile step (round-2 Critical #2). Where Pester is absent (host mode / fresh local clone / non-container CI), `test:unit` **skips with an actionable message** rather than hard-failing the composite (round-3 #7) — so `validate.ps1`'s "runs identically on host and container" property holds; the autopilot container always has it.
- **Composite gate lives in `package.json`, not `.autopilot.json`** (round-3 Critical #1). The host launcher validates `.autopilot.json` `test` against a `StartsWith` allowlist that rejects `&&` and any non-`npm test` prefix ([launch.ps1](scripts/autopilot/launch.ps1), [autopilot.schema.json](schemas/autopilot.schema.json)), so a composite there can never launch; and having the agent rewrite `.autopilot.json` mid-run would un-fix the "fixed evidence-runner" premise. So `.autopilot.json` `test` stays the allowlist-clean `npm test`, and **`package.json`'s `test` script** is the composite — `npm run validate-plan && npm run test:unit && pwsh -NoProfile -File scripts/validate.ps1` (invoking `validate.ps1` directly, not `npm test`, to avoid infinite recursion). The agent never edits config; **Absolute Rule 5 is amended** only to bless the unchanged, allowlist-validated `npm test` as the committed evidence-runner (config-defined, never parsed from plan text). REQ-24 reconciles this in the design notes.
- **Verification + receipt + archival gate.** `ci`/autopilot verify `test:`/`file:`/`review:` evidence at phase/plan crosschecks (not per-save) and write `✓/✗ REQ-N — <evidence> — <result> — <commit>` to plan-folder `evidence.md`. The gate is enforced **in `autopilot.agent.md` itself** (round-3 #3): the agent's "On Plan Completion" block reads `evidence.md` and **refuses to archive/PR** on any `✗`/unrun REQ unless deferred in Decisions — not only in the `ci` skill asset (autopilot never loads ci assets), since today the agent archives unconditionally.

### Conditional `@human` finalization (auto-decided, PR-safe)

- **Distinct from mid-plan block** (round-2 Critical #3). Today exit 42 = "@human step hit mid-phase, stop"; the entrypoint pushes a branch with **no PR**, and archive/PR happen only in the agent's "On Plan Completion" block (which does *not* run on exit 42). The finalization path therefore: (1) verifies evidence + assesses harvest; (2) if all green and unambiguous → completes autonomously (archive + PR as today); (3) if a decision is needed → **pushes the branch, creates a draft PR (`@human` label), writes the marker, then exits 42**.
- **Finalization is carved out of Absolute Rule 7** (round-3 #6). An ordinary `@human` step makes the agent commit + report + exit 42 *immediately*; the standard conditional Finalization step must instead run verify → push → draft-PR → marker *before* exiting 42. Step 4.3 amends the agent's step-7 classification / Rule 7 to special-case the Finalization step distinctly from a mid-plan `@human` — otherwise the agent stops before creating the PR, re-introducing the round-2 #3 bug.
- **Marker status + ordering defined** (round-3 #9). `.autopilot-finalize-needed` is an **uncommitted, gitignored** repo-root file (it must survive to the entrypoint without polluting the PR/branch). Explicit sequence: commit progress → **push branch** → `gh pr create --draft` → write marker → exit 42 (the remote head must exist before `gh pr create --head`).
- **Container-mode scope** (round-3 #8). Conditional finalization is wired into `container-entrypoint.sh`, which maps the marker to "PR created, human finalization requested". This plan scopes conditional finalization to **container-autopilot** (the plan's own execution mode); `launch-host.ps1` host mode does not support it — stated in REQ-21 so a host run isn't silently mis-handled.

### Bloat + skill structure

- **Thin-orchestrator + assets.** `cip`/`ci` `SKILL.md` become short step-sequence orchestrators reading `assets/*.md` at the relevant stage. `cip`: `interview-guide.md`, `drafting-guide.md`, `dr-guide.md`. `ci`: `execution-guide.md`, `crosscheck-guide.md`. **Preservation checklist** maps each existing `cip` Step-4 rule (specificity, simplest-first, security-by-design, architecture-lock-in, cross-ref integrity) to an asset section, and a non-negotiable summary stays inline (round-1 #12).
- **State anchor.** `cip` writes/updates `<!-- cip-stage: ... -->` per stage; `ci` always emits its progress snapshot.
- **Size limits.** Warn @ 20KB/400 lines, block @ 35KB/700 lines. Concise steps; multi-paragraph rationale → `decisions/<topic>.md`. DR stays 3 rounds; CR cadence unchanged.

### Misc

- **Lightweight review wins only.** `cr` default diff scope = changed files **including committed branch deltas** (round-2 #19); parallel subagent dispatch asserted once (already implemented — round-1 #18). Deep redesign deferred.
- **Phase budget** advisory: `S=1`/`M=2`/`L=3`, cap 6/phase, overridable; `Test-Plan.ps1` warns only.
- **Registry.** New assets added to owning `plugin.json` `files[]`; registry rebuilt; dogfood byte-equivalence re-verified. Validator + fixtures are repo infra, not plugin payload.
- **Roles:** all `@ai-agent` except the conditional finalization escalation. **Execution-mode:** container-autopilot, phase scope (dogfoods autopilot).
